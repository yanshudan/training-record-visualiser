Training Record Visualiser

A single-page React app to plan, run, and visualise strength training. React 18ViteMUI rechartsAzure Blob

Multi-user by design. All data is keyed by userId and stored as full-overwrite JSON blobs under users/{userId}/…. Run instantly in local or SAS mode; switch on the backend for real multi-user.

Features

  1. Today (Start your plan) — generates today's targets from history + progressive-overload rules, runs the workout with per-set & rest timers, lets you slide reps/weight when a target is missed, add exercises in place, and saves the session.
  2. Plan (Configure) — CRUD over exercises (body part, unit, defaults) and the ordered progressive-overload rules (e.g. +1 rep each time, +2.5 kg when all sets hit 12, reset to 10).
  3. Timetable — define a daily schedule, visualised as a 24-hour clock or a vertical flow (switchable).
  4. Stats — body-part frequency, volume-over-time per body part, and per-exercise progression (top weight, estimated 1RM, volume), all driven by your own exercise definitions.

Quick start

npm install
npm run dev      # http://localhost:3123

The app starts in local storage mode (browser only) and migrates any legacy localStorage["trainingRecords"] data automatically.

Azure Blob: versioning & bug-tolerance

Each save is a full overwrite of records.json. To stay bug-tolerant:

  • Blob Versioning — every overwrite auto-creates a recoverable previous version.
  • Soft Delete — recover from accidental deletes/overwrites within a retention window.
  • ETag / If-Match — optimistic concurrency prevents lost updates (the app reloads on conflict).
az storage account blob-service-properties update \
  --account-name <acct> --enable-versioning true \
  --enable-delete-retention true --delete-retention-days 30

Blob layout

users/{userId}/records.json     # full accumulating history (full overwrite each save)
users/{userId}/exercises.json   # exercise defs + overload rules
users/{userId}/timetable.json   # daily schedule

Data migration plan

  1. On first load, runMigrations() checks for a v2 records.json.
  2. If absent and legacy localStorage["trainingRecords"] exists, it is parsed by the legacy serializer and written as v2; a flag prevents re-running.
  3. Default exercise definitions are seeded if none exist.
  4. Future schema bumps: read schemaVersion and transform forward in the same place.
